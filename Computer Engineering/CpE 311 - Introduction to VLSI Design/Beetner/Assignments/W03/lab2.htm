CpE 311 Lab Exercise 2

VHDL Based FPGA Design

Due: Thursday, March 6.

Purpose

The purpose of this exercise is to become familiar with VHDL based design flow for Actel Field Programmable Gate Arrays (FPGA).  In this lab you will design a medium scale circuit using VHDL and logic synthesis in order to form a basis for comparison to schematic based design. You will use the same specifications as you used in Lab 1. 

Problem Description

Repeat lab 1 but use VHDL to capture your design instead of a schematic.  Implement your registered ALU with an Actel 40MX02 FPGA. The steps you need to perform to complete this lab are outlined in Act1 Synthesis Example.  In summary, do the following:

·      Read the hints at the end of this report and skim through the Act 1 Synthesis Example

·      Write a .vhd file describing your design

·      Compile with vcom

·      Do a functional simulation using vsim.  Use the testbench provided, but also perform your own checks.  Print out the list output from the testbench showing all of the entity port signals and any other internal signals you think are relevant.

·      Synthesize your VHDL code using Examplar’s Leonardo (See Act1 Synthesis Example).  Record the number of modules used.

·      Place and route logic modules within the FPGA using Actel Designer.  Record the worst case propagation delays.

·      Perform a timing simulation with vsim, using the timing parameters produced by designer.

Do NOT simply convert your lab 1 schematic design to a VHDL structural netlist (Mentor includes tools to do this).  Organize your model in the form of separate combinational logic and sequential logic processes.  You should at least create an entity for the alu and an entity for each register (4-bit accumulator and 1-bit carry).  Use the following entity for your top-level design (the registered alu):

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;
entity ralu is
 port(B: in std_logic_vector(3 downto 0); -- external operand
      S: in std_logic_vector(1 downto 0); -- function code
      RST: in std_logic;                  -- reset (active high)
      Sysclk: in std_logic;               -- system clock
      ACC: out std_logic_vector(3 downto 0); -- accumulator out
      CY: out std_logic);                 -- carry out bit
end entity ralu;

 

If the entity is not specified in the above manner, then the testbench will not work.

Hints

 

·      The “Act1 Synthesis Example” is very helpful… I would at least glance over it over before doing anything else.

·      Use the scripts given for Leonardo in the Act1 Synthesis Example!  You can perform the same operations by hitting buttons, etc, in Leonardo but it is very easy to forget something or get something wrong and come up with a design that seems to compile but doesn’t quite work in the end.  I have seen several students have problems because they didn’t use the script.

·      Testbench.  Be sure to analyze your .vhd file with vcom before you analyze the testbench.  Since the testbench will instantiate your alu design, your design should already exist.  The testbench will take about 850 ns to run and will announce “test completed” when it is done.

·      Use DT analyze to find the maximum delay between registers and the critical path – Not Spectrum.  To get to dtanalyze, type designer at the unix prompt, load your design as in lab 1, compile it, lay it out (hit the Layout button), then hit DT Analyze.  Choose your desired parameters and hit OK (be sure to specify a source and sink – start and finish point).  The table shows you the delays in descending order.  If you select one and hit “Expand”, it will give you details about the delay path.

·      If you’d like to “see” what your analyzed circuit looks like, you may use Leonardo to do so.  As presented in the slides given in class, type “leonardo” at the command prompt, select the appropriate technology and your .vhd file and then press “run flow” (this may take a minute or two).  Under the “Tools” menu item, you may view the RTL or gate-level synthesized version of your VHDL code.  This is a good way to attempt to “correlate”  worst case propagation delays with VHDL source code.

·      To run the testbench while using your timing parameters, use the command:

vsim –sdfmax /dut=hdl/alu.sdf ralu_tb a1

where alu.syn should be replaced by the name of the sdf file created for your design. This command will specify that the sdf timing parameters are applied only to the entity instance named “dut” (which is your alu in this case) within the testbench entity.

·      The pin definition for pins ripped from vectors (busses) might look something like:

DEF entity_name

PIN a<1>;

             PIN: 11.

PIN a<2>;

             PIN 12.

etc.

where a<1> is bit one of vector a.
